# Shop Simulators Project Guide

This file is the working guide for building future games on the Shop Simulators site.

The site idea:
- A static HTML/CSS/JS website made of many small shop/business simulator games.
- Each game is a different shop, store, or small business.
- The core emotion is the pleasure of making money, improving a business, and watching income grow.
- The experience should be simple enough to play casually, but tangible enough to feel like real shop work.
- The site should stay compatible with Cloudflare Pages/free static hosting and future AdSense approval.

The current reference game is Coffee Shop Simulator.

## Core Promise

Every game should make the player feel:

"I started with a tiny business, served customers, earned money, spent it wisely, and made the shop better."

The money loop is the heart of the site. Cash gain, tips, upgrades, and visual shop growth must be clear and satisfying.

## Standard Game Loop

Use this loop for every shop unless there is a strong reason not to:

1. Customer enters the shop.
2. Customer joins a visible line or waiting area.
3. Customer shows an order/request.
4. Player selects or prepares the requested product/service using visible shop objects.
5. Player confirms completion with a natural object, such as a bell, register, counter, tray, ticket, scanner, or station.
6. Game checks accuracy and timing.
7. Correct service pays money, maybe tips, and improves reputation/streak.
8. Wrong/late service causes no sale, refund, missed customer, or reputation loss.
9. Player spends money on stock and upgrades.
10. The shop visually improves and earns faster.

Do not make "serve" the whole action. The player should do at least one physical-feeling step before the final confirmation.

## Money Feeling

Money is the main emotional reward. It must never feel hidden.

Required patterns:
- Cash should be large, readable, and attractive.
- Money popups should appear near the register/counter/customer action, not randomly in the screen center.
- Positive feedback should be instant: sound, popup, customer reaction, stat update.
- Tips and streaks should feel like bonuses.
- Reputation should be visible but secondary to cash.
- Upgrades should cost visible money and make future money feel faster.

Good HUD pattern:
- Desktop: a compact but attractive in-scene cash board/register display, usually near the top-left or another natural shop object.
- Mobile: keep cash and reputation visible at the top of the game screen.
- Cash should be emphasized more than profit/served/rep.

## Page Layout Pattern

Game pages should be toy-first.

Current rule:
- The game page is the playable experience.
- Do not put SEO/content sections below the game page.
- Do not put a persistent game list beside gameplay.
- Do not put close right/left ad boxes directly beside the game scene.
- Keep quiet far-left and far-right browser gutters for future AdSense side rails.
- Homepage handles game discovery.
- Future article/info pages can handle SEO text if needed.

Desktop game page:

Header / logo / small navigation
Centered game screen
Quiet far-left and far-right page space for future side rails
No below-game content
No game-page footer required

Current Coffee Shop desktop pattern:
- Header remains compact.
- Centered game canvas is large, but not full browser width.
- Side gutters remain visible for future side-rail ads.
- Game canvas uses the art's natural aspect ratio when possible.
- Coffee Shop uses a 1586 / 992 ratio because that matches the cafe background.
- Avoid using an arbitrary 16:9 crop if it cuts off important art.

Desktop width guidance:
- Homepage grid: about 1180px max width.
- Game pages: should usually be larger than the old early-MVP 1120px idea when the art benefits from it.
- Use a centered game area around 1300-1500px max when the art benefits from it.
- Preserve side gutters using viewport-based constraints.
- Example pattern: centered game width up to about 1500px, with left/right gutters kept quiet.

Mobile game page:
- The game can scroll if needed.
- Keep all controls inside the game screen.
- Avoid tiny controls.
- If object-pinned controls collide on mobile, convert them into compact in-scene panels.

## In-Scene UI Pattern

All core game controls should live inside the shop scene and should feel like shop objects.

Good examples:
- Shelf products are clickable products on actual shelves.
- Restock actions are crates, boxes, bags, or backroom supplies.
- Orders are receipts, tickets, speech bubbles, menus, or customer request cards.
- Final confirmation is a bell, register, scanner, counter button, oven door, pickup window, or service station.
- Upgrades are icons or props connected to actual shop improvements.
- Save/pause/reset can be small corner utility buttons.

Avoid:
- Large detached management panels that cover beautiful background art.
- Generic buttons that say "Serve" as the main interaction.
- Cards floating over the shop when they could be placed on shelves, counters, chalkboards, receipts, or registers.
- UI that blocks the customer line, shop floor, or main product/service area.

## Asset Direction

The assets must harmonize with the game's main background art.

For each game:
1. Choose the primary background/scene style first.
2. Match all product, customer, tray, station, button, and HUD assets to that style.
3. Do not mix polished 3D assets, flat vector icons, and pixel backgrounds unless intentionally stylized.
4. Prefer custom local assets when a third-party pack does not match.
5. Use `image-rendering: pixelated` or `crisp-edges` for pixel-style sprites.
6. Keep important assets transparent where they sit on top of the scene.

Coffee Shop current pattern:
- Main background is generated pixel cafe art.
- Product and bell assets use custom pixel-style SVGs in `assets/img/coffee-shop/pixel-products/`.
- Customer tokens are CSS-built but blocky/pixel-styled.
- Tray, tickets, stock counts, and money HUD use blocky/pixel-compatible styling.
- Older smoother SVG product assets remain available, but the pixel-products set is preferred for Coffee Shop.

Asset credits:
- Track all production assets in `assets/ASSET_CREDITS.md`.
- For third-party assets, record asset name, creator, source URL, license, attribution requirement, and where it is used.
- CC0/public domain is safest.
- Kenney is a strong first source because many packs are CC0.
- OpenGameArt and itch.io are useful, but licenses vary and must be checked per asset.

Important asset lesson:
- A license-safe asset is not automatically a good asset.
- If it does not match the game art, make a custom local sprite or choose a different pack.

## Gameplay Mechanics Pattern

Each game should have:

- Visible customer demand.
- Visible product/service supply.
- A preparation step.
- A confirmation step.
- Correct/wrong outcome.
- Money reward.
- Reputation or satisfaction.
- Restocking or operating costs.
- Upgrades that change both numbers and visuals.
- Save, pause, and reset controls.

Coffee Shop reference implementation:
- `state.activeCustomerId` tracks the current front customer.
- `state.tray` tracks the prepared product.
- Products are picked from shelf buttons.
- The service bell validates the tray against the active order.
- Wrong products cause no sale and reputation loss.
- Correct products give cash, tips, streak, and happy customer feedback.
- Restock crates refill supplies.
- Pause freezes spawning, patience timers, barista automation, and autosave ticking.
- Save uses localStorage only.

Future games should copy the concept, not necessarily the exact names.

Example translations:
- Bakery: pick bread/cake from display, place on tray, ring register.
- Grocery: pick requested item from shelf, scan at register.
- Pet Shop: pick pet food/toy, bring to counter, ring sale.
- Flower Shop: assemble bouquet, wrap it, hand to customer.
- Pizza Shop: choose pizza, bake/finish it, serve at counter.
- Bookstore: find book, place at checkout, ring register.

## Customer Pattern

Customers should be visible, not just rows in a list.

Required:
- Customers enter from a door or edge.
- Customers form a real line or wait at a visible spot.
- The first customer is clearly the active/front customer.
- Customers show a request via bubble, ticket, or order card.
- Customers have patience.
- Customers react happy/annoyed.
- Customers leave after service or after losing patience.

Nice to have:
- Customer archetypes with small differences.
- Student: waits longer, tips less.
- Office worker: impatient, tips more.
- Regular: better streak/tip behavior.
- Reviewer: affects reputation more.
- Tourist: normal behavior, flavor text.

## Upgrade Pattern

Every upgrade should change the room if possible.

Good upgrade design:
- It unlocks a product/service or improves speed/income.
- It has a visible prop in the shop.
- It makes the business feel more successful.

Coffee Shop examples:
- Espresso Machine unlocks latte and appears on counter.
- Pastry Display unlocks cake and appears beside the register.
- Outdoor Sign increases foot traffic and appears near the door.
- Second Grinder increases value and appears near the counter/shelf.
- Barista Helper can auto-serve and appears as a worker.
- Cozy Seating increases patience and improves tables/seating.
- Loyalty App improves tips and appears near register.
- Nitro Tap unlocks cold brew and appears behind counter.

Rule:
- If an upgrade cannot be represented visually, consider delaying it or redesigning it.

## Stock And Restocking Pattern

Stock should be spatial and visual.

Good:
- Product availability shown on shelves or displays.
- Stock count badges can exist, but the object should still feel physical.
- Low stock changes count/color or shelf state.
- Restocking uses crates, boxes, supply bags, delivery shelf, backroom area, or vendor cart.

Avoid:
- Only showing inventory as abstract numbers.
- Restock buttons that feel detached from the shop.

## Pause / Save Pattern

Use localStorage for MVP saves.

Current rule:
- No backend.
- No login.
- No account.
- Save progress in browser localStorage.
- Include Reset Shop.
- Include Pause.
- Export/import can come later.

Pause behavior:
- Stop customer timers.
- Stop new customer spawning.
- Stop automation.
- Stop autosave ticking.
- Keep save/reset UI responsive.
- Do not allow free serving/restocking/upgrading while paused.

## Monetization Pattern

The game should feel fullscreen/toy-like even if ads are used later.

Current rule:
- Real ads are disabled until AdSense approval.
- Vignettes are intentionally disabled.
- Do not place banner ads inside the game screen.
- Do not add below-game ad/content blocks to game pages.
- Desktop side rails are the preferred future ad format.
- Keep far-left and far-right gutters quiet.
- Natural future H5 ad breaks can happen after milestones, shift summaries, or several completed orders.
- Never interrupt during a drag, rapid click, order prep, or first interaction.

## Homepage Pattern

Homepage is the discovery page.

Use:
- Main game grid.
- Clear shop/game names.
- Cute visual cards.
- Short descriptions.
- Links into each game.

Homepage can carry more discovery/SEO text than game pages.

## Future SEO Pattern

Do not force SEO text below every game page.

Better options:
- Homepage intro/discovery copy.
- Separate article/info pages later, such as "Best shop simulator games" or "How to play shop simulator games."
- Game pages should stay focused on the toy.

## Visual Style Rules

Use these rules for future games:

- Match all UI and sprites to the main background style.
- If background is pixel art, use pixel-style assets and blocky UI.
- If background is vector/cartoon, use vector/cartoon assets consistently.
- If background is realistic photo, avoid cartoon products unless intentionally playful.
- Cash HUD must be readable and appealing.
- Keep important art visible.
- Put controls on natural shop surfaces when possible.
- Keep left/right gutters quiet for future ads.
- Avoid one-note palettes; let each shop have its own theme.
- Text inside small UI must fit.
- Use stable dimensions for buttons, products, tickets, counters, trays, and HUD items.

## Build Pattern For A New Game

Recommended order:

1. Read this guide.
2. Choose the shop concept and money fantasy.
3. Choose the main art style/background.
4. Decide the natural object locations: shelves, counter, register, door, waiting line, upgrade area.
5. Build the page shell using the current game-page pattern.
6. Build the gameplay state: money, reputation, supplies, queue, active customer, prepared item/tray, upgrades.
7. Implement customer spawning and visible line.
8. Implement product/service picking.
9. Implement final confirmation and accuracy check.
10. Implement money popups, tips, wrong-order feedback, and customer reactions.
11. Implement restocking and upgrades.
12. Make upgrades visually change the shop.
13. Add save, pause, reset.
14. Tune HUD readability.
15. Run syntax, DOM, asset-reference, and visual checks.

## Testing Checklist

Before considering a game page done:

- JavaScript parses.
- All `getElementById` IDs exist in HTML.
- Referenced assets exist.
- CSS has balanced braces.
- Game works after refresh.
- Save works.
- Pause stops timers and automation.
- Reset works.
- Customer line appears.
- Product picking works.
- Wrong product fails correctly.
- Correct product pays correctly.
- Money display is obvious.
- Reputation display is readable.
- Upgrades can be bought when affordable.
- Restocking works when affordable.
- Game screen does not cover future side-rail gutters on desktop.
- Mobile layout is usable.
- Art style is consistent.
- Asset credits are updated.

## Current Best Recommendation

Build each game as a static, one-screen, shop-specific toy:

- Static site on Cloudflare Pages.
- One domain.
- Many game URLs.
- No backend for MVP.
- Save with localStorage.
- Game pages are one-screen toys.
- Homepage handles discovery.
- Future side-rail ads only after approval/testing.
- Use consistent assets per game.
- Make money earning visible, satisfying, and central.
